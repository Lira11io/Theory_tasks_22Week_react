  1. Что такое props и можно ли использовать props в функциональных компонентах?

    Props представляет коллекцию значений, которые ассоциированы с компонентом. Эти значения позволяют создавать динамические компоненты, которые не зависят от жестко закодированных статических данных.
    Их можно использовать как в классовых, так и в функциональных компонентах. В функциональном компоненте props записываются в качестве параметра в функцию и инкапсулируют свойства объекта.
    Класс компонента также извне получает объект свойств, который доступен через this.props.

  2. Нужно ли выделять в отдельный компонент статью в блоге?

    Для удобства, да. Общий вид статьи можно сохранить в компоненте, и потом наполнять разным содержанием. Мне кажется, что можно даже выделить в отдельные компоненты составные части статьи.

  3. Можно ли использовать React без JSX?

    JSX не является обязательным для работы с React. React можно использовать без JSX. Это особенно удобно, когда вы не хотите настраивать транспиляцию в процессе сборки. Всё, что вы можете сделать при помощи JSX, может быть сделано на чистом JavaScript.

  4. Чем отличается JSX от HTML?

    Хотя JSX и пытается быть похожим на HTML, у них все же есть некоторые отличия.

    • В JSX все свойства DOM и атрибуты (включая обработчики событий) должны быть записаны в camelCase. Например, атрибут tabindex превращается в tabIndex. Исключением являются атрибуты aria- и data-, они записываются точно так же, как и в обычном HTML.
    • Так как for — зарезервированное слово в JS, в React-элементах используется свойство htmlFor.
    • Вместо атрибута class в JSX используется имя свойства в DOM: className.
    • Так как это похожий на XML синтаксис, одиночные теги в JSX должны быть закрыты.
    • Обычный HTML не очень безопасен. Любой текст, который должен оставаться текстом, необходимо экранировать перед выводом. Иначе если внутри содержится HTML, то он будет проинтерпретирован. Ситуация может стать опасной, если этот текст на сайт добавляют сами пользователи. JSX работает по-другому. Всё, что выводится обычным способом, безопасно по умолчанию и экранируется автоматически. А вот в тех местах, где этого не требуется, экранирование отключается свойством dangerouslySetInnerHTML. В это свойство передаётся объект со свойством __html, значением которого является строка с HTML.
    • Имена CSS-свойств также должны быть записаны в стиле camelCase.

  5. Для чего нам нужны свойства (props) компонентов?

    Они позволяют не писать новый компонент каждый раз при необходимости заменить даже небольшую часть объекта (например, карточки товара), и использовать компонент, подставляя нужные данные.

  6. В примере с CardList чем можно было бы заменить ?

    Можно использовать сокращенный синтаксис и обернуть код в пустые скобки "< >"...">", однако, нужно быть внимательным, так как его понимают не все инструменты. Или использовать обычный div, но это приведет к добавлению лишних узлов в DOM.

  7. Можно ли сказать, что классовые и функциональные компоненты равнозначны по функциональности?

    С точки зрения React эти два компонента эквивалентны. Ну почти. Некоторые до сих пор считают, что: «Применение классов позволяет пользоваться большим количеством возможностей компонентов, например — состоянием». Однако, с появлением хуков (hooks) это уже не так.
    Кстати, даже без использования хуков, у функциональных компонентов даже есть преимущество перед классовыми. Функциональные захватывают значения свойств, а классовые при использовании, к примеру, отложенной функции из-за this.props могут их потерять.
